# 状态栏命令执行功能设计

## 目标

让 DevDock 像常见 macOS 应用一样出现在顶部状态栏中，用户可以从状态栏菜单直接执行已注册命令。

## 范围

- 应用启动时创建 DevDock 状态栏图标。
- 状态栏菜单展示当前已注册命令。
- 点击命令菜单项后在后台执行该命令。
- 命令成功时不弹出主窗口。
- 命令失败时弹出并聚焦主窗口，显示执行日志和失败结果。
- 菜单提供“打开 DevDock”“刷新命令”“退出 DevDock”操作。
- 注册或删除命令后刷新状态栏菜单。

不支持从状态栏输入任意 shell 命令、不支持传参、不支持并发队列、不支持终止运行中命令。

## 推荐设计

采用 Tauri 2 的 tray icon API。Rust 启动阶段创建状态栏图标和菜单，菜单项 id 使用稳定前缀映射到命令名。点击命令菜单项后，后端复用现有 `run_registered_command_streaming_in_dir()` 执行 DevDock 管理的入口文件。

前端新增托盘执行事件监听：

- `command-run-started`：把命令模块切到前台状态并初始化日志面板，但不主动弹出窗口。
- `command-output`：沿用现有实时输出事件。
- `command-run-finished`：更新最终退出码和成功/失败状态。
- `command-run-failed`：更新错误状态。

后端只有在命令退出码非 0 或执行过程报错时调用窗口显示逻辑。这样状态栏执行成功时保持后台安静，失败时用户能立即看到原因。

## 错误处理

- 命令不存在或不是 DevDock 管理入口：后端发出失败事件并弹出主窗口。
- 命令退出码非 0：后端发出完成事件，弹出主窗口，前端展示失败状态。
- 菜单刷新失败：保留基础菜单，并通过后端错误信息避免影响应用启动。

## 安全影响

状态栏菜单只能执行 DevDock 已注册命令，仍然沿用命令名校验、入口路径定位和 DevDock 管理标记检查。不开放任意命令字符串，也不把菜单文本直接传给 shell。

## 验证

- Rust 单元测试覆盖非 0 退出码需要弹窗、0 退出码不需要弹窗的判断。
